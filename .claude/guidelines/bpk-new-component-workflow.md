# Backpack New Component Workflow

Read this file when scaffolding a **new component package** (`packages/bpk-component-*`), adding a new **V2/Ark UI compound** under an existing package, or any time you create a new `*.module.scss` file. The items below are checks that lint/test discovers slowly — knowing them up front saves a full lint loop each.

## 1. Ark UI components require an ESLint allowlist entry

The repo bans `className` on most React components via `@skyscanner/rules/forbid-component-props`. Wrappers around Ark UI primitives (`Collapsible.Root`, `Checkbox.Root`, `SegmentGroup.Item`, …) must be added to the rule's `allowedFor` list.

When you wrap a new Ark UI component, add every part you pass `className` to:

```jsonc
// .eslintrc → "@skyscanner/rules/forbid-component-props" → forbid[0].allowedFor
"Collapsible.Root",      // allowed for styling to be passed to ark-ui component
"Collapsible.Trigger",   // allowed for styling to be passed to ark-ui component
"Collapsible.Indicator", // allowed for styling to be passed to ark-ui component
"Collapsible.Content",   // allowed for styling to be passed to ark-ui component
```

Use the same trailing comment style as the existing entries — they are grouped by primitive.

## 2. SCSS modules need a sibling `.module.css` for Jest

Every `*.module.scss` has a sibling `*.module.css` consumed by Jest at test time. The `.module.css` files are **gitignored** (`*.css` in `.gitignore`) — they are build artefacts, not source. After creating or editing a `.module.scss`, regenerate the sibling locally so `cssModules()` returns the right class names:

```bash
node -e "require('./scripts/scss/style-compiler.js').compile('./packages/bpk-component-<name>/src/<File>.module.scss')"
```

Forgetting this step means tests run against stale styles and snapshot tests render with no classes applied. The full repo build (`pnpm run build:sass`) does the same thing for every `.scss` file. Do **not** stage `.module.css` files for commit — they will be regenerated by CI.

## 3. CSS class modifiers are kebab-case — convert prop values explicitly

`cssModules` does not transform names. A prop value like `style="onContrast"` will not match a SCSS modifier `--on-contrast`; it will look for `--onContrast`. Convert in the component:

```ts
const styleModifier = style === 'onContrast' ? 'on-contrast' : 'default';
const className = getClassName('bpk-collapsible', `bpk-collapsible--${styleModifier}`);
```

Keep SCSS modifiers in kebab-case (matches the rest of the codebase and stylelint's `selector-class-pattern`).

## 4. Avoid prop names that collide with HTML reserved attributes

ESLint rules in this repo treat any JSX prop named `style`, `className`, or `key` as if it were the DOM attribute. A custom prop named `style` (e.g. `style: 'default' | 'onContrast'`) will trip `react/style-prop-object` on every consumer site and require an inline `eslint-disable-next-line` everywhere it is used.

When designing a new component API, prefer `variant`, `surface`, `tone`, or `appearance` over `style`. If a spec hands you a `style` prop name, push back before implementation rather than littering the codebase with disables.

## 5. The `backpack/use-tokens` rule rejects `'inherit'`

The "unknown colour" lint rule treats any string in a `color`/`background` position that isn't a foundation token as a brand violation — including the CSS keyword `'inherit'`. Don't write `style={{ color: 'inherit' }}` in stories or components. Either:

- Drop the inline style and rely on the CSS cascade (the parent's `color:` will inherit naturally on text nodes), or
- Set the colour to a real token via SCSS.

## 6. Stylelint enforces property order and `@media` placement

Two rules consistently fail on first write:

- `order/properties-order` — `display` before `width/height` before `justify-content` before `align-items`, etc. Run `npx stylelint <file> --fix` to auto-resolve.
- `order/order` — `@media` blocks must come **before** nested `&:hover`/`&[data-state=…]` rules inside the same selector, not after.

Run `npx stylelint 'packages/bpk-component-<name>/**/*.scss' --fix` before recompiling the `.module.css`.

## 7. Ark UI's `data-state` has three values, not two

For animated primitives (Collapsible, Dialog, Popover) Ark exposes `data-state="open" | "closed" | "closing"`. Tests that toggle open → closed in quick succession can race with the closing animation in jsdom. Prefer asserting via `onOpenChange` callback firing, or use `defaultOpen` to test the closed-from-open path separately.

For exit animations, target **both `closing` and `closed`** in your CSS — Ark passes through `closing` while the animation runs and lands on `closed` once it finishes. Targeting only `closed` skips the visible animation:

```scss
&[data-state='closing'],
&[data-state='closed'] {
  animation: bpk-collapse $duration ease;
}
```

## 8. Reapply native HTML attributes when Ark only emits `data-*` ones

Zag (the state machine behind Ark UI) is intentionally style-system-agnostic. For some primitives — `Collapsible.Trigger`, for example — it only sets `data-disabled` / `data-state` and does **not** emit the native `disabled` / `aria-disabled` attribute. The button stays tabbable and is announced as enabled even though clicks are no-op'd.

Bpk wrappers must reapply the native attribute. The pattern is to read state from Ark's hook and forward it explicitly:

```tsx
import { Collapsible, useCollapsibleContext } from '@ark-ui/react';

const BpkCollapsibleTrigger = (...) => {
  const { disabled } = useCollapsibleContext();
  return <Collapsible.Trigger disabled={disabled} ... />;
};
```

The same applies to `:focus-visible` — Zag does not always expose `data-focus-visible` (it does for some primitives, not for `Collapsible.Trigger`). Default to the standard CSS pseudo-class for focus styling:

```scss
.bpk-foo__trigger:focus-visible {
  @include utils.bpk-focus-indicator;
}
```

## 9. Bpk components do not accept `className` or `style` from consumers

Backpack components own their surface, focus, animation, RTL, and accessibility behaviour. Allowing consumers to pass `className` or `style` would let them silently break those guarantees. **Do not declare `className?: string` or `style?: CSSProperties` in your prop types.** The internal `className` you forward to Ark/Chakra/Radix primitives is fine — that's what the ESLint allowlist in §1 covers.

If a consumer needs to position or constrain the component, they wrap it in `BpkBox` / `BpkFlex` / `BpkVStack` (see `bpk-layout-components.md`). State this explicitly in the component README so reviewers don't push back asking for the prop.

The single exception is `BpkVessel`, which is the legacy migration escape hatch and accepts `className` / `style` deliberately.

## 10. Stylelint auto-converts physical CSS properties to logical ones

The repo enforces RTL-safe CSS via stylelint. When you write physical sizing properties they are auto-converted on `--fix`:

- `width` → `inline-size`
- `height` → `block-size`
- `min-width` → `min-inline-size`
- `min-height` → `min-block-size`
- `padding-left` / `padding-right` → `padding-inline-start` / `padding-inline-end`
- `margin-left` / `margin-right` → `margin-inline-start` / `margin-inline-end`
- `text-align: left` / `right` → `text-align: start` / `end`
- `border-left` / `border-right` → `border-inline-start` / `border-inline-end`

You can write the physical form on first pass; `npx stylelint --fix` will rewrite it. Don't fight the conversion by manually re-writing logical → physical — it will be reverted on the next lint run and breaks RTL.

## When drafting the spec issue (not the implementation)

If you're writing the GitHub issue for a future component, lock these in before opening it — they are cheap at spec time and expensive at implementation time:

- **API prop names** must avoid HTML-attribute collisions (`style`, `className`, `key`, `id`, `ref`, `for`). Prefer `variant`, `surface`, `tone`, or `appearance`.
- **Anatomy** for compound components must list the exact Ark UI part each Bpk part wraps (`BpkFoo.Trigger` → `Foo.Trigger`). This is what determines the ESLint allowlist additions in §1.
- **Package layout decision** must be explicit: new package (`packages/bpk-component-foo/`) vs. nested under an existing v1 (`packages/bpk-component-bar/src/BpkBarV2/`).
- **Variants list** should describe each variant's surface/text token mapping, not just hex values from Figma — so the implementer doesn't have to round-trip through `bpk-token-value-lookup.md`.
- **Animation contract** should name the Ark/Zag CSS variables to bind to (e.g. `--height` on `Collapsible.Content`) so the implementer knows whether to use keyframes, transitions, or AnimateHeight.
- **Do not propose `className` or `style` props on Bpk parts** — Backpack owns the styling. If consumers need to wrap the component in a layout, they use `BpkBox`/`BpkFlex` etc. around it. See §9.

## Checklist when scaffolding a new component package

```
packages/bpk-component-<name>/
├── index.ts                         # default export + named type exports
├── README.md                        # follow bpk-component-checkbox/README.md shape
└── src/
    ├── Bpk<Name>.tsx                # compound export object (if applicable)
    ├── Bpk<Name>Root.tsx            # one file per part
    ├── Bpk<Name>.module.scss        # source — sibling .module.css is gitignored, see step 2
    ├── Bpk<Name>-test.tsx
    ├── accessibility-test.tsx       # jest-axe
    └── Bpk<Name>.stories.tsx
```

Then in order: `npx eslint <pkg> --ext .ts,.tsx` → `npx stylelint '<pkg>/**/*.scss' --fix` → recompile SCSS → `TZ=Etc/UTC npx jest <pkg>` → `pnpm run check-bpk-dependencies` → `pnpm run check-react-versions`.
