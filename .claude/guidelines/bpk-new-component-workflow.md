# Backpack New Component Workflow

Read this file when scaffolding a **new component package** (`packages/bpk-component-*`), adding a new **V2/Ark UI compound** under an existing package, or any time you create a new `*.module.scss` file. The items below are checks that lint/test discovers slowly ‚Äî knowing them up front saves a full lint loop each.

## 1. Ark UI components require an ESLint allowlist entry

The repo bans `className` on most React components via `@skyscanner/rules/forbid-component-props`. Wrappers around Ark UI primitives (`Collapsible.Root`, `Checkbox.Root`, `SegmentGroup.Item`, ‚Ä¶) must be added to the rule's `allowedFor` list.

When you wrap a new Ark UI component, add every part you pass `className` to:

```jsonc
// .eslintrc ‚Üí "@skyscanner/rules/forbid-component-props" ‚Üí forbid[0].allowedFor
"Collapsible.Root",      // allowed for styling to be passed to ark-ui component
"Collapsible.Trigger",   // allowed for styling to be passed to ark-ui component
"Collapsible.Indicator", // allowed for styling to be passed to ark-ui component
"Collapsible.Content",   // allowed for styling to be passed to ark-ui component
```

Use the same trailing comment style as the existing entries ‚Äî they are grouped by primitive.

## 2. SCSS modules need a sibling `.module.css` for Jest

Every `*.module.scss` has a sibling `*.module.css` consumed by Jest at test time. The `.module.css` files are **gitignored** (`*.css` in `.gitignore`) ‚Äî they are build artefacts, not source. After creating or editing a `.module.scss`, regenerate the sibling locally so `cssModules()` returns the right class names:

```bash
node -e "require('./scripts/scss/style-compiler.js').compile('./packages/bpk-component-<name>/src/<File>.module.scss')"
```

Forgetting this step means tests run against stale styles and snapshot tests render with no classes applied. The full repo build (`npm run build:sass`) does the same thing for every `.scss` file. Do **not** stage `.module.css` files for commit ‚Äî they will be regenerated by CI.

## 3. CSS class modifiers are kebab-case ‚Äî convert prop values explicitly

`cssModules` does not transform names. A prop value like `style="onContrast"` will not match a SCSS modifier `--on-contrast`; it will look for `--onContrast`. Convert in the component:

```ts
const styleModifier = style === 'onContrast' ? 'on-contrast' : 'default';
const className = getClassName('bpk-collapsible', `bpk-collapsible--${styleModifier}`);
```

Keep SCSS modifiers in kebab-case (matches the rest of the codebase and stylelint's `selector-class-pattern`).

## 4. Avoid prop names that collide with HTML reserved attributes

ESLint rules in this repo treat any JSX prop named `style`, `className`, or `key` as if it were the DOM attribute. A custom prop named `style` (e.g. `style: 'default' | 'onContrast'`) will trip `react/style-prop-object` on every consumer site and require an inline `eslint-disable-next-line` everywhere it is used.

When designing a new component API, prefer `variant`, `surface`, `tone`, or `appearance` over `style`. If a spec hands you a `style` prop name, push back before implementation rather than littering the codebase with disables.

## 5. The `backpack/use-tokens` rule rejects `'inherit'`

The "unknown colour" lint rule treats any string in a `color`/`background` position that isn't a foundation token as a brand violation ‚Äî including the CSS keyword `'inherit'`. Don't write `style={{ color: 'inherit' }}` in stories or components. Either:

- Drop the inline style and rely on the CSS cascade (the parent's `color:` will inherit naturally on text nodes), or
- Set the colour to a real token via SCSS.

## 6. Stylelint enforces property order and `@media` placement

Two rules consistently fail on first write:

- `order/properties-order` ‚Äî `display` before `width/height` before `justify-content` before `align-items`, etc. Run `npx stylelint <file> --fix` to auto-resolve.
- `order/order` ‚Äî `@media` blocks must come **before** nested `&:hover`/`&[data-state=‚Ä¶]` rules inside the same selector, not after.

Run `npx stylelint 'packages/bpk-component-<name>/**/*.scss' --fix` before recompiling the `.module.css`.

## 7. Ark UI's `data-state` has three values, not two

For animated primitives (Collapsible, Dialog, Popover) Ark exposes `data-state="open" | "closed" | "closing"`. Tests that toggle open ‚Üí closed in quick succession can race with the closing animation in jsdom. Prefer asserting via `onOpenChange` callback firing, or use `defaultOpen` to test the closed-from-open path separately.

## When drafting the spec issue (not the implementation)

If you're writing the GitHub issue for a future component, lock these in before opening it ‚Äî they are cheap at spec time and expensive at implementation time:

- **API prop names** must avoid HTML-attribute collisions (`style`, `className`, `key`, `id`, `ref`, `for`). Prefer `variant`, `surface`, `tone`, or `appearance`.
- **Anatomy** for compound components must list the exact Ark UI part each Bpk part wraps (`BpkFoo.Trigger` ‚Üí `Foo.Trigger`). This is what determines the ESLint allowlist additions in ¬ß1.
- **Package layout decision** must be explicit: new package (`packages/bpk-component-foo/`) vs. nested under an existing v1 (`packages/bpk-component-bar/src/BpkBarV2/`).
- **Variants list** should describe each variant's surface/text token mapping, not just hex values from Figma ‚Äî so the implementer doesn't have to round-trip through `bpk-token-value-lookup.md`.
- **Animation contract** should name the Ark/Zag CSS variables to bind to (e.g. `--height` on `Collapsible.Content`) so the implementer knows whether to use keyframes, transitions, or AnimateHeight.
